## 1. Данные и предобработка

### 1.1 Исходный массив

Выданный корпус — **1453 файла, ~4.9 ГБ**: PDF (1163, часть — сканы), DOCX/DOC (133),
ZIP/RAR (95, с вложениями и многотомниками), XLS (49), PPTX (5). Языки — русский,
английский, отчасти китайский/испанский. Материал разнородный: обзоры, статьи, доклады,
журналы 2015–2025, материалы конференций (Copper, ALTA, New Caledonia, МПГ, Кобальт),
книги, рыночные отчёты (CRU/GFMS/Mitsui), патенты.

### 1.2 Обработка волнами

Корпус обрабатывался не одним проходом, а **волнами** по приоритету содержательной
ценности — это позволило поднять end-to-end MVP на первой волне и итеративно
наращивать покрытие:

| Волна | Содержание | Файлов (ok) |
|---|---|---|
| 1 | Обзоры + Статьи + Доклады | 186 |
| 2 | Журналы 2015–2025 | 217 |
| 3 | Материалы конференций + книги | 69 |
| 4 | Рыночные отчёты, старые журналы, архивы | остаток |

Снимок после волн 1–3 (`corpus/README.md`): **472 документа status=ok, 52 352 чанка,
~43 млн токенов, 0 failed**. По разделам: Журналы 217, Обзоры 110, Материалы
конференций 69, Статьи 60, Доклады 16. Волна 4 нарастила корпус до **1320 документов /
116 801 чанка** (снимок `documents.jsonl`/`chunks.jsonl` на вечер 04.07; манифест — 1322
записи) и продолжает дописываться.

Пайплайн идемпотентен (инкрементальность по хэшу файла), повторный прогон ничего не
дублирует; статусы `ok/failed/skipped` фиксируются в манифесте.

### 1.3 Извлечение текста и OCR-стратегия

Извлечение по форматам: PyMuPDF (PDF с текстовым слоем), python-docx / LibreOffice-fallback
(doc/docx), python-pptx (pptx), openpyxl/xlrd (xls → текстовые таблицы).

**OCR — постранично, а не по документам.** Разведка выборки (200 из 1161 PDF, зафиксировано
в PLAN §«Результаты разведки»): полных сканов в корпусе **~0 %**, у ~8 % PDF есть отдельные
«пустые» страницы (обложки, диаграммы). Поэтому OCR применяется точечно к страницам без
текстового слоя, а не к целым файлам — это на порядок сократило OCR-бэклог.

OCR-движок — **docling** (0.6–1.2 с/стр, стабильный Markdown). Альтернатива **marker**
исключена после замера: обрыв на 30 с из-за прокси-таймаута, пустой ответ. Локальный
fallback — tesseract (rus+eng). По методам извлечения (снимок волн 1–3): `ocr_docling` 229,
`docx` 120, `pymupdf` 110, `doc` 7, `pptx` 5, `xlsx` 1.

### 1.4 Чанкинг и его баг-фикс

Стратегия: чанки ~1000 токенов, overlap 100, по границам абзацев/заголовков; язык
определяется на чанк; фильтруется мусор (колонтитулы, оглавления, реклама).

**Баг «56-КБ чанков».** На плотных числовых таблицах и слитных PDF ранняя версия чанкера
эмитила аномально большие чанки (десятки КБ). Последствия: (а) такой текст ронял целый
батч Yandex-эмбеддера в HTTP 400 («отравление» батча); (б) числовые значения размывались
на слишком большом окне. Фикс — **жёсткий кап** в чанкере (`chunker.py`: `HARD CAP: never
emit a chunk above MAX_CHUNK_CHARS`, `_split_oversized`), одноразовая утилита
`fixup_oversized.py` (порог `max_chars=6000`) и **перечанковка 295 документов**. На стороне
эмбеддингов добавлена дополнительная защита: усечение до 1800 символов + per-text изоляция
сбоя батча (см. §4).

### 1.5 Граф знаний (продукт предобработки)

Извлечение знаний — двухслойное: детерминированный rule-слой (числа, единицы, компараторы,
диапазоны, даты, гео) поверх LLM-слоя (сущности, связи, Assertion с дословной цитатой).
Итог по волнам 1–2 (PLAN §6-а): **граф 28 159 узлов / 47 722 ребра, 88 % Assertion с
дословной цитатой-основанием**. Реестр концептов — **183 концепта / 421 алиас** RU/EN.
Валидатор дословности **отбросил 9 % чисел**, предложенных LLM (число, не найденное дословно
в чанке, в граф не попадает). Экстракция продолжается на восстановленном ключе: на вечер
04.07 обработано **54 391 чанк из 600 документов** (`pipeline/extract/extractions.sqlite`).

**Граф провалидирован SHACL** (`docs/ontology.ttl` + `docs/shapes.ttl`, отчёт
`docs/VALIDATION_REPORT.md`): реальный прогон pyshacl по всему графу (28 159 узлов + 47 722
рёбер = 722 300 RDF-триплетов) — 1437 находок (256 Violation / 1181 Warning), это честная
картина шума извлечения, не подгонка. Провенанс факт-рёбер (`source_doc`, `confidence`,
`method`, `extracted_at`) и структура Assertion (statement/confidence/review_status/evidence)
проходят валидацию на подавляющем большинстве элементов; основной остаточный шум —
неканонические операторы условий (`≥`, `min`, `gt`…) и качественные value вместо чисел —
зафиксирован как бэклог нормализации rule-слоя.

## 2. Задача и метрики

### 2.1 Постановка

Retrieval-augmented QA поверх научно-технического корпуса: пользователь задаёт вопрос на
естественном языке (RU/EN), система возвращает структурированный ответ **только на основе
найденных доказательств**, с цитатами на конкретные документы/фрагменты, проверяемыми
числами, а также подсветкой противоречий и «пустых зон». Ключевое требование заказчика —
**верифицируемость** (каждый факт — со ссылкой) и **честность** (не галлюцинировать по темам
вне корпуса).

### 2.2 Метрики (харнесс `qa/harness.py`)

| Метрика | Определение |
|---|---|
| **retrieval hit-rate** | доля запросов с `must_mention_docs`, где среди `citations` найден ≥1 ожидаемый документ (матч по подстроке `filename + title`) |
| **doc recall (avg)** | средняя доля найденных ожидаемых паттернов документов |
| **number accuracy** | доля запросов с `must_contain_numbers`, где ВСЕ числа присутствуют в ответе/цитатах (толерантно к `,`/`.`; эхо запроса вырезается) |
| **number recall (avg)** | средняя доля найденных чисел |
| **citation-rate** | доля запросов с `must_have_citations`, где есть ≥1 цитата |
| **honesty-rate** | на adversarial (`expect_empty=true`): доля запросов, где система НЕ вернула цитат (любая цитата на тему вне корпуса = галлюцинация) |
| **latency p50/p95/max** | round-trip wall-clock по всем прогонам |

### 2.3 Устройство eval-set (`qa/eval_set.yaml`)

**35 размеченных запросов** (45 прогонов с учётом EN-вариантов), покрывающих 6 классов:
`lookup` (13), `review` (8), `compare` (8), `aggregate` (6), `gap` (3), `adversarial` (7).
Включает 4 golden-запроса из ТЗ (обессоливание воды ОФ; циркуляция католита при ЭЭ никеля;
распределение Au/Ag/МПГ штейн/шлак; закачка шахтных вод РФ vs зарубеж).

**Ревизия 2 — методологически ключевой шаг.** Первая ревизия паттернов была составлена под
FIXTURE-заголовки. При переходе на реальный корпус (403 документа) паттерны и числа выверены
заново: имена файлов — по `corpus/documents.jsonl` (реальные `CM_06_15.pdf`, `ЦМ № 09-23.pdf`,
`Обзор…никеля и меди.docx`), а числа — **дословным grep по `corpus/chunks.jsonl`** (числа из
fixture-легенды «1,2 м³/ч; 92,5 %» заменены на реально существующие: 1250–1350 °C в d000097;
1,9/4,5/5,8 м³/ч в d000381; 67,1/25,1 мг/л в d000275). Паттерны добавлялись только для
смыслово-релевантных документов, а не «под выдачу ретривала» — иначе метрика меряла бы саму
себя.

**Почему важен adversarial-класс.** Заказчик прямо требует не «прятать» отсутствие данных и
не галлюцинировать. Adversarial-запросы — темы, заведомо отсутствующие или лишь доменно-смежные
(алюминий, метод Чохральского, скотоводство, виноделие, LLM). Honesty-rate на них — прямой
прокси бизнес-риска «система выдумала факт со ссылкой». Именно доменно-смежные adversarial
(алюминий-электролиз против никель-электролиза) оказались самыми трудными: их не разделяют ни
BM25, ни семантика, ни граф — потребовался отдельный многоуровневый гейт честности (§4.3).

### 3.1 Ключевые сюжеты улучшений (нарратив)

1. **Гейт честности на fixture: 28.6 % → 100 %.** Baseline шаблонного синтеза галлюцинировал
   по adversarial. Введён скор-гейтинг (§4.3) — honesty вышла на 100 % на fixture (стр. 0→3).

2. **Переход на реальный корпус: golden сломались → диагностика → рекалибровка.** Замена
   fixture-графа реальным (стр. 5–6) обрушила honesty (42.9 %) и hit (13.5 %): реальное
   распределение эмбеддингов и лексики другое. Диагностика (`backend/README` §«Retrieval на
   реальном корпусе») показала, что корпус реально содержит доменно-смежный контент
   (удельный расход энергии на выплавку алюминия — d000342; метод Чохральского — d000294),
   неотделимый ни лексикой, ни семантикой. Ответ — переход в doc-doc пространство,
   прекомпьют эмбеддингов и трёхуровневый гейт → honesty 100 %, hit восстановлен (стр. 7–8).

3. **Number accuracy 11 % → 66.7 %.** После ревизии чисел на реальные значения точность
   упала до 11.1 % (стр. 9): значения часто лежат в СОСЕДНЕМ чанке относительно
   процитированного или за границей 400-символьной цитаты. Фикс — три механизма (§4.2):
   ES number-boost, numeric-neighbor enrichment («сосед-чанк»), doc-name ветка →
   **66.7 %** (стр. 11), цель ≥60 % достигнута.

4. **Баг чанкера (56-КБ чанки) → жёсткий кап → перечанковка 295 док.** (см. §1.4).

5. **Hit-rate 43 → 54 → 86.5 → 94.6.** Включая методологически важную ревизию eval-паттернов
   (стр. 8→9): без правки ядра, лишь синхронизацией разметки с реальными doc_id, — иллюстрация
   принципа «сначала убедись, что метрика измеряет то, что нужно».

6. **Исследование эмбеддинг-моделей: e5 отклонён → Qwen3-0.6B.** По итогам исследования для
   локального GPU-сервиса (`deploy/embed_service/server.py`) выбран `Qwen/Qwen3-Embedding-0.6B`
   (Apache-2.0, **32K контекст**, 1024-dim, Matryoshka 32..1024): у `multilingual-e5-large`
   лимит **512 токенов**, а наши чанки в среднем **~933 токена (p90 1092)** — e5 молча резал бы
   половину чанка (`backend/README.md` §«Пространства»). Полный корпус пересчитан **локально
   на GPU заказчика (RTX 4070)**: **113 589 чанков × 1024-dim** (`meta.json`, src=real) — без
   расхода облачных квот. Кросс-язычный EN→RU матчинг подтверждён eval-прогонами (EN-варианты
   golden проходят hit, лог `harness_qwen_yandex.log`). Важная находка калибровки: у Qwen на
   однородном металлургическом корпусе косинусы «сжаты» (on-topic ~0.42–0.54 vs same-domain
   ~0.32), поэтому пороги sem-гейта — **пер-пространственные** (`retrieval.py::_SPACE_GATES`:
   Qwen 0.42/0.55/0.38 против Yandex 0.58/0.66/0.50). Результат A/B: строки 16–17 таблицы —
   number accuracy 77.8 % (лучший результат по числам) при honesty 100 %.

7. **Инцидент «смерть ключа» → мультипровайдерность за часы (04.07).** Грант фолдера Yandex
   Cloud исчерпан (~35M токенов) — 403 на все вызовы. Деградацию поймал харнесс, а не жалобы
   вручную: строка 13 — number accuracy 0 %, синтез упал в детерминированный шаблонный
   фоллбэк (система не легла — «мягкая деградация» отработала как задумано). Ответ —
   мультипровайдерный шлюз `shared/llm_gateway.py`: боевой инцидент-конфиг на **Groq**
   (планер `llama-3.1-8b-instant` — 14 400 req/день, 6000 TPM, native json_schema; синтез
   `gpt-oss-120b` — 1000 req/день, **8000 TPM**: пригоден для интерактива, непригоден для
   батч-экстракции), фолбэк-цепочка `gigachat → openrouter → openai`. Задокументированные
   «грабли»: `qwen3-32b` сжигает max_tokens на `<think>`-рассуждения; маркеры цитат `【n】`
   у gpt-oss нормализуются в `[n]`; GigaChat подключён (OAuth, авто-рефреш токена), но
   B2B-ключ без купленных пакетов даёт `402 Payment Required`. После восстановления
   YC-ключа синтез возвращён на `yandex-lite` (строка 16) — латентность вернулась в норму,
   а мультипровайдерность осталась страховкой и аргументом ресурсо-оптимальности.

## 4. Архитектурные решения и обоснование

### 4.1 Гибридный retrieval + RRF

Три ветки параллельно: **(A)** Elasticsearch bool+range по чанкам (лексика RU+EN морфология,
фасеты, числовые range); **(B)** векторный поиск по прекомпьют-эмбеддингам чанков в **doc-doc
пространстве** (находка: query↔chunk в doc-doc разделяет лучше, чем query-space); **(C)**
Cypher-anchor по концептам графа. Слияние — **RRF (k=60)** поверх нормализованных веток,
цитаты диверсифицируются по документам (лучший чанк на уникальный doc_id), подграф ответа
ограничен ≤60 узлами.

*Trade-off.* Neo4j vector-index и ES `dense_vector` рассмотрены и отклонены: matmul `q·M` по
~29k×256 в памяти занимает <10 мс, не требует переиндексации и подхватывает растущий `.npy`
по mtime без рестарта — дешевле в реализации без проигрыша в латентности.

### 4.2 Rule-first числа (три механизма)

1. **Rule-first экстракция**: числа берёт детерминированный слой (87 тестов правил в
   `pipeline/extract/rules`); валидатор дословности отбросил 9 % чисел LLM. Гарантия: число,
   отсутствующее дословно в источнике, в граф не попадает.
2. **ES number-boost**: should-клаузы `match_phrase` по нормализованным вариантам чисел
   («1,2»/«1.2», с единицей и без), boost 3.0.
3. **Numeric-neighbor enrichment** («сосед-чанк»): для топ-цитат собираются тексты-кандидаты
   (свой чанк целиком; соседи seq±1; чанки conditions-индекса по параметрам интента), из них
   выбираются числовые окна ПО КЛЮЧЕВЫМ СЛОВАМ запроса — фикс кейса «правильный документ,
   значение в соседнем чанке» (f34: цитируется c0006, а «1250–1350 °C» в c0007).

### 4.3 Assertion-модель и гейт честности

**Assertion-центричная модель**: факты — не «голые рёбра», а утверждения с evidence
(дословная цитата), confidence, review_status, версионированием (`supersedes`). Противоречия
кластеризуются и помечаются `contradicts`/`disputed` — не прячутся.

**Гейт честности — три уровня, детерминированный где возможно:**
1. **Скор-гейт** на чанк: сигналы `lex` (ES-хит ≥12 % от топа), `sem` (doc-doc cosine ≥ 0.58),
   `concept` (документ привязан к концепт-якорю / отличительный концепт в тексте). Проходит
   при ≥2 сигналах ИЛИ `cos≥0.66` ИЛИ `concept & cos≥0.50`. Пусто → честный ответ «не найдено».
2. **Foreign-material rule** (~3 мс, без LLM): запрос о материале вне домена (алюминий) и без
   core-материала → честный пустой ответ, инфлекс-толерантный точный матчинг.
3. **LLM-judge backstop** (`judge_relevance`, yandexgpt-lite, ~20 токенов) — вызывается ТОЛЬКО
   когда ни одна цитата не привязана к отличительному материалу запроса (подозрительный
   пропуск); in-domain golden его не триггерят → нет ложных отказов и накладной латентности.
   Fail-open при недоступности LLM.

### 4.4 Мультипровайдерность (LLM и эмбеддинги) как архитектурный принцип

Решение не привязано ни к одному облаку — оба «горячих» слоя вынесены за шлюзы:

- **LLM** (`shared/llm_gateway.py`): бэкенды `yandex` (делегирование штатному клиенту,
  байт-в-байт совместимость), `openai_compatible` (единый код для OpenRouter / OpenAI /
  Groq / vLLM / Ollama / LM Studio), `gigachat` (OAuth с авто-рефрешем токена, structured
  output в prompt-режиме), `mock`. Пер-роль модели (`LLM_MODEL_PLANNER/SYNTHESIS/…`),
  фолбэк-цепочки (`LLM_FALLBACK`), пер-провайдерный прокси (`GROQ_PROXY`), учёт токенов
  `by_provider` в `/api/health`. Смена модели = смена `.env`, без правки кода. Ценность
  доказана инцидентом 04.07 (§3.1 п.7).
- **Эмбеддинги** (`shared/embeddings_gateway.py`): понятие **«пространства»** —
  `{space_id, provider, model, dim, prefix_scheme, query_kind}` с каталогом
  `graph/embeddings/{space_id}/` и версионируемым прекомпьютом. Готовые пространства:
  `yandex-256` (облако, 256-dim) и `qwen3-0.6b` (**локальный GPU-сервис**
  `deploy/embed_service/`, 1024-dim, полный корпус пересчитан на RTX 4070 заказчика
  бесплатно). Пороги гейта — пер-пространство (косинус-распределения разные). Подключение
  новой модели — 3 шага без изменения кода retrieval.

Это одновременно ответ на критерий Q&A «при прочих равных выигрывает ресурсо-оптимальное
решение»: тяжёлый пересчёт — на своём GPU, интерактив — на самой дешёвой достаточной модели.

### 4.5 Trade-off «точность ↔ скорость» (обоснование по Q&A)

Требование 3–5 с заказчик обозначил как нефункциональное («целиться, но не упарываться»),
допустив на Q&A №3 «размен точность ↔ скорость/стоимость, если он оправдан ростом качества».
Реализованные разумные размены:
- **Синтез по умолчанию `yandexgpt-lite`, не Pro.** У Pro «пол» латентности ~5 с независимо
  от длины ответа, что ломает p95≤5 с; lite даёт так же грунтованный ответ за ~2.5 с
  (переключатель `SCITANGLE_SYNTH_MODEL=pro` для max-качества).
- **Прекомпьют эмбеддингов вместо on-the-fly.** On-the-fly эмбеддинг кандидат-чанков был
  «убийцей p95» (стр. 6 — p95 20.8 с); косинус берётся из `.npy`.
- **LLM-judge только на подозрительных пропусках** — не на каждом запросе.

Итог референс-конфигурации (стр. 11): p95 **5.3 с** при hit 94.6 % / honesty 100 %.

## 5. Воспроизводимость

- **Детерминизм.** Устранён недетерминизм f04: стабильные tie-break по `chunk_id` во всех
  сортировках (RRF-ветки + финал) — при частичном прекомпьюте много чанков с cosine=0.0,
  раньше порядок решал dict-order и цитаты «плавали» между прогонами. Пороги гейта — константы
  в `retrieval.py`. Числа в планере всегда детерминированы (регекс). При недоступности
  ключа/сети всё деградирует в детерминированные фоллбэки — система не падает.
- **Чекпойнт-методология «тест на 1/3».** Дорогие джобы (экстракция, прекомпьют) —
  резюмируемые с чекпойнтами; harness замерялся на промежуточных состояниях корпуса (снимки
  285/360/385/403 док в отчётах) — качество контролировалось до полного прогона.
- **Харнесс одной командой.** `.venv-f/bin/python qa/harness.py` → отчёт `md`+`json`
  (`summary`, `by_class`, `rows`) для трендов/CI. Нагрузочный — `qa/load_test.py --n 20`.
  Каждый отчёт фиксирует условия замера (`backend health`: размер корпуса/графа, провайдер,
  разбивка токенов `by_provider`) — прогоны сопоставимы постфактум.
- **Тесты.** 87 тестов rule-слоя; **27 pytest бэкенда** (интеграционные, RBAC-матрица,
  trace-панель «как получен ответ», upload); **18 тестов LLM-шлюза** на mock-провайдере без
  живых ключей (structured-output фолбэки, фолбэк-цепочки, пер-провайдерный прокси,
  OAuth-рефреш GigaChat); e2e Playwright — 19 сценариев на live-профиле (в т.ч. upload,
  adversarial-честность, explain-панель; скриншоты в `e2e/screenshots/`).
- **SHACL-валидация графа.** `backend/validate_graph.py` — воспроизводимый прогон pyshacl по
  `docs/shapes.ttl`/`docs/ontology.ttl` на живом графе; отчёт `docs/VALIDATION_REPORT.md`.
- **Docker.** `docker compose up -d` (Neo4j+ES) для dev; `deploy/docker-compose.prod.yml`
  (Neo4j+ES+API+UI+nginx) для стенда — жюри открывает готовые контейнеры без сборки; HTTPS
  через Caddy-sidecar (`deploy/Caddyfile`, https://158-255-4-151.sslip.io).
- **Смена LLM-провайдера и эмбеддинг-модели конфигом.** `shared/llm_gateway.py` +
  `shared/embeddings_gateway.py` (§4.4): Yandex / Groq / GigaChat / OpenRouter / OpenAI /
  локальные — сменой `.env`; эмбеддинг-«пространства» версионируются в
  `graph/embeddings/{space}/meta.json` (модель, dim, дата, покрытие, src=real|hash).

## 6. Ограничения и дальнейшая работа

Честный список того, что не доведено до идеала:

1. **Number accuracy 77.8 %, не 100 %.** Лучший замер (стр. 17, Qwen-пространство) выше цели
   ≥60 %, но часть числовых кейсов не проходит: остаточный пробел — значения, вытесняемые из
   топ-6 цитат (f02: «1,9» из d000381). Направление — реранкер числовых окон и расширение
   conditions-индекса.
2. **Экстракция покрывает не весь корпус.** На вечер 04.07 в граф извлечено 54 391 чанк из
   **600 документов** при корпусе 1320 — экстракция продолжается на восстановленном YC-ключе
   (до этого блокер: грант исчерпан, ~35M токенов, 403 на всё). Eval-паттерны настроены под
   403-док срез — hit на растущем корпусе (86.5 %) занижен относительно референса 94.6 %;
   после доэкстракции нужна ре-калибровка eval под полный корпус.
3. **Реранкер — в roadmap.** Сейчас ранжирование — RRF без cross-encoder реранкера
   (кандидат — `Qwen/Qwen3-Reranker-0.6B`, влезает в тот же GPU); даст прирост и на hit, и
   на числах, но добавляет латентность — включать под замер p95. Там же — второе гибридное
   пространство `bge-m3` (dense+sparse), добавляется как ещё одно `Space` без переделки.
4. **Latency под контеншеном.** При РАБОТАЮЩЕЙ фоновой экстракции 429/backoff поднимают p95 до
   ~13 с (общая квота 10 сессий на ключ) — тяжёлые джобы держатся на паузе во время демо/замера.
   Митигировано частично собственным инференсом эмбеддингов; для LLM — фолбэк-цепочка.
5. **Honesty на доменно-смежном adversarial частично опирается на LLM-judge** — при полной
   недоступности LLM backstop работает fail-open (пропускает), детерминированы только уровни
   1–2 гейта.
6. **Внешние провайдеры под лимитами не тянут интерактив в бюджете** (стр. 14–15: p95 30–46 с
   на Groq/OpenAI-цепочке) — это инцидент-режим; целевая конфигурация — yandex-lite или
   собственный vLLM.
