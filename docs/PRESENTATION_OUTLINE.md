Сценарий защиты по «философии Максима» (продуктовое визионёрство): не отчёт по ТЗ, а история
о том, как решена бизнес-проблема управления знаниями. ~5–7 слайдов ядра + демо-нить.
Живое демо часто «зависает» → основа показа — **скринкаст**; скриншоты — из `e2e/screenshots/`.

## Слайд 1 — Боль учёного

- **Мысль:** 90+ лет знаний на 4 языках; поиск нужного = «ручной квест по архивам». Учёный
  тратит на поиск больше времени, чем на анализ; память компании теряется, эксперименты
  дублируются.
- **Визуал:** до/после — гора PDF/архивов vs один NL-запрос → ответ со ссылками.
- **Цитата заказчика:** «нужна не БД, а живая карта знаний».

- **Визуал:** скриншот главного экрана поиска — `e2e/screenshots/search-result-live.png`.

## Слайд 3 — ДЕМО-НИТЬ (сердце презентации, показывать в скринкасте)

Единый сквозной сценарий на одном golden-запросе — показать всю ценность за 60–90 секунд:

1. **Golden query** — ввод: «Циркуляция католита при электроэкстракции никеля: оптимальная
   скорость потока» (f02) → evidence packet с ответом.
   - Скринкаст: печать запроса → появление ответа. Скрин: `search-result-live.png`.
2. **Верификация цифры в первоисточнике** — клик по цитате `[n]` → выезжает дословный фрагмент
   документа с подсвеченным числом (скорость м³/ч / температура).
   - Скринкаст: клик → drawer. Скрин: `search-source-drawer-live.png`. **← это кульминация
     «верифицируемости», задержаться здесь.**
3. **Противоречие** — показать блок «Противоречия» / красное ребро `contradicts` на графе
   (источник A vs источник B) — система не прячет расхождение.
   - Скринкаст: переход на граф. Скрин: `graph-live.png`.
4. **Пустая зона** — gap-запрос (напр. озонирование сточных вод) → честный ответ «данных нет /
   пробел», а не галлюцинация. Подчеркнуть: **honesty на adversarial 100 %**.
   - Скринкаст: ввод заведомо отсутствующей темы → честный пустой ответ.
5. **Добавление нового документа** — загрузка файла → извлечение → появление в графе/поиске.
   Демонстрация масштабируемости на новых данных (важный критерий заказчика).
   - Скринкаст: upload → повторный поиск находит новый источник.

## Слайд 4 — Как это работает (1 слайд архитектуры)

- **Мысль:** ingest (OCR постранично, чанкинг) → извлечение (rule-first числа + LLM Assertion
  с дословной цитатой) → гибридный retrieval (Neo4j + ES, RRF) → гейт честности → синтез
  только из доказательств.
- **Визуал:** диаграмма из `README.md` (блоки A→B→C→D) — упростить до одного экрана.
- **Одна фраза-«крючок»:** «LLM синтезирует, но НЕ выдумывает: число без дословного источника
  в ответ не попадает».

## Слайд 5 — ML-культура (метрики-итерации) [1 слайд, при запасе — 2]

- **Мысль:** решение не «случайное на GPT» — оно измерялось и улучшалось (16 замеренных
  итераций). Показать **таблицу итераций** из `docs/TECH_REPORT.md §3` (сжато): baseline →
  реальный корпус → инцидент ключа → мультипровайдерность → Qwen-эмбеддинги.
- **Ключевые цифры:** hit-rate **43 → 94.6 %** (референс), honesty **28.6 → 100 %**,
  number accuracy **11 → 77.8 %** (актуальный прогон на Qwen-пространстве), p95 5.3–9.9 с.
- **История-иллюстрация (выбрать 1–2):**
  - «eval был потолком, не ретривал»: ревизия eval-паттернов подняла hit без правки ядра —
    показывает зрелость методологии (сначала проверь, что метрика измеряет то, что нужно).
  - Number accuracy 11 % → 77.8 % через «сосед-чанк»-фикс + Qwen-эмбеддинги.
  - **Инцидент «смерть ключа»**: грант YC исчерпан (403) → деградацию поймал ХАРНЕСС (num
    accuracy 0 %, но система не легла — мягкая деградация) → мультипровайдерный шлюз за
    часы → Groq на время инцидента → возврат на yandex-lite. Живая иллюстрация и
    ML-культуры, и отказоустойчивости.
- **Визуал:** таблица итераций + подпись «полный тех-репорт: `docs/TECH_REPORT.md`».

## Слайд 6 — Масштабирование в enterprise (change management) [Максим это любит]

- **Мысль:** технология — 20 %, внедрение — 80 %. Сложность масштабирования AI в enterprise:
  доверие пользователей (отсюда верифицируемость, honesty и панель «Как получен ответ» как
  продуктовые требования, а не «фича»), качество данных (rule-first + провенанс + SHACL-
  валидация онтологии), онтология как расширяемый контракт, человек-в-контуре (ручная правка
  графа экспертом).
- **Сильный тезис «не привязаны ни к одному облаку»** (ресурсо-оптимальность — критерий Q&A):
  LLM-шлюз (Yandex / Groq / GigaChat / OpenRouter / OpenAI / локальный vLLM) и
  эмбеддинг-«пространства» меняются конфигом; полный корпус эмбеддингов (113 589 × 1024)
  пересчитан на GPU заказчика бесплатно. Провайдер-инцидент пережит без остановки продукта.
- **Мысль-мостик:** продукт спроектирован под рост — обработка волнами, идемпотентный пайплайн,
  upload новых документов end-to-end, готовые docker-контейнеры + HTTPS-стенд.

## Слайд 7 — Roadmap

- Cross-encoder **реранкер** (`Qwen3-Reranker-0.6B` — влезает в тот же GPU) + второе
  гибридное пространство `bge-m3` (прирост hit и числовой точности под замер p95).
- Доэкстракция полного корпуса (идёт на восстановленном ключе) + ре-калибровка eval.
- Расширение онтологии и доменов; обработка таблиц/графиков.
- Внешние источники (Crossref/OpenAlex/ORCID); версионирование фактов; датчики установок.
