# Contract — `retrieval_trace` (объяснимость поиска)

Панель «как система нашла ответ» строится фронтендом (агент UX-Explainability) по полю
`retrieval_trace` ответа `POST /api/search`. Поле объясняет, ПОЧЕМУ выбраны именно эти
источники (какие ветки ретривала сработали, какие концепты распознаны, прошёл ли
honesty-гейт) и служит второму критерию жюри — верифицируемости.

Владелец контракта: агент Answer-Quality (`backend/app/search.py::_build_retrieval_trace`).
Ядро ретривала (`backend/app/retrieval.py`) НЕ меняется — трейс собирается из того, что
`retrieve()` уже возвращает (пер-цитатные сигналы + агрегатные счётчики гейта).

## Расположение

```
POST /api/search  →  SearchResponse.retrieval_trace  (nullable)
```

`null` допустим только для устаревших кэш-записей; свежие ответы всегда содержат объект.

## Схема

```jsonc
retrieval_trace: {
  "branches": [
    {
      "name": "lexical" | "semantic" | "graph" | "doc-name",
      "method": "человекочитаемое описание ветки",   // напр. "BM25 (Elasticsearch)"
      "n_candidates": number | null,                 // см. «Ограничения»: сейчас всегда null
      "n_passed_gate": number | null,                // сколько ИТОГОВЫХ цитат несёт сигнал ветки
      "active": boolean,                             // только для doc-name: сработала ли ветка
      "top_signals": [
        { "doc_id": "d000057", "title": "Методы очистки шахтных вод", "signal": "cos=0.71" }
      ]
    }
  ],
  "concepts_matched": [
    { "concept_id": "c_mine_water" | null, "name": "шахтные воды", "matched_from": "запрос" }
  ],
  "gate": {
    "passed": boolean,          // true = найдены доказательства (citations непусты)
    "reason": "текст",          // почему гейт пропустил/отклонил
    "n_candidates": number,     // кандидатов-чанков рассмотрено до honesty-гейта
    "n_kept": number,           // чанков прошло гейт
    "namedoc": boolean          // было сильное совпадение имени/заголовка документа
  },
  "docs_considered": number     // = gate.n_candidates (чанков-кандидатов рассмотрено)
}
```

## Ветки (`branches`)

Четыре ветки ретривала, в порядке отображения:

| `name`     | Что означает | `signal` в top_signals |
|------------|--------------|------------------------|
| `lexical`  | лексический матч BM25 в Elasticsearch по тексту чанка | `rrf=<score>` (RRF-вес слияния) |
| `semantic` | косинус запрос↔чанк в активном эмбеддинг-пространстве (прекомпьют) | `cos=<cosine>` |
| `graph`    | документ связан с концепт-якорной сущностью графа ИЛИ отличительный термин запроса найден в тексте | `<doc_id>` |
| `doc-name` | сильное совпадение имени/заголовка документа с запросом | — (см. `active`) |

`n_passed_gate` = число ИТОГОВЫХ цитат, у которых сработал сигнал этой ветки (одна цитата
может засчитаться нескольким веткам — сигналы не взаимоисключающие; именно ≥2 сигналов —
основной критерий honesty-гейта). `top_signals` — до 3 сильнейших цитат ветки для превью.

Ветка `doc-name` несёт только глобальный флаг `active` (какие именно документы совпали по
имени, ядро не выдаёт пер-цитатно), поэтому её `n_passed_gate` = `null` при `active=true`.

## Концепты (`concepts_matched`)

Распознанные в запросе концепты (материалы/процессы/установки) — привязка ответа к онтологии.
Берутся из концепт-сущностей графа (`concept_id` = id узла) либо, если граф не дал якорей, из
концептов планировщика (`concept_id=null`). `matched_from` сейчас всегда `"запрос"` (концепт
извлечён из текста запроса); поле зарезервировано под будущие источники (напр. `"граф"`).

## Гейт (`gate`)

`passed=true`, если ответ опирается на источники. `reason` — человекочитаемое объяснение:
- пропуск: `"<N> чанков прошли скор-гейт (≥2 сигналов или сильный семантический)"`;
- отклонение: `"скор-гейт: доказательств недостаточно (…)"` → фронт показывает честный
  «не найдено» и не рисует ложных источников.

## Ограничения (честно для жюри по ML-культуре)

- **`branches[].n_candidates` сейчас всегда `null`.** Пер-веточные счётчики кандидатов ДО гейта
  живут внутри `retrieval.py` (ветка эмбеддингов, зона другого агента) и наружу не отдаются.
  Трейс считает статистику по ВЫЖИВШИМ (пост-гейт) цитатам, а размер пула до гейта отражён
  агрегатом `docs_considered` / `gate.n_candidates`. Когда ядро начнёт отдавать пер-веточные
  счётчики — заполнить это поле без изменения фронта (форма стабильна).
- `docs_considered` = число чанков-кандидатов (не документов) — исторически так называется
  счётчик гейта; фронту показывать как «рассмотрено фрагментов».

```jsonc
"retrieval_trace": {
  "branches": [
    {"name": "lexical", "method": "BM25 (Elasticsearch)", "n_candidates": null,
     "n_passed_gate": 4, "top_signals": [
        {"doc_id": "d000057", "title": "Методы очистки шахтных вод", "signal": "rrf=0.0312"}]},
    {"name": "semantic", "method": "cosine query↔chunk (precompute)", "n_candidates": null,
     "n_passed_gate": 3, "top_signals": [
        {"doc_id": "d000057", "title": "Методы очистки шахтных вод", "signal": "cos=0.71"}]},
    {"name": "graph", "method": "концепт-якорь графа / термин в тексте", "n_candidates": null,
     "n_passed_gate": 5, "top_signals": [
        {"doc_id": "d000275", "title": "CM_05_15", "signal": "d000275"}]},
    {"name": "doc-name", "method": "совпадение имени/заголовка документа", "n_candidates": null,
     "active": true, "n_passed_gate": null, "top_signals": []}
  ],
  "concepts_matched": [
    {"concept_id": "c_mine_water", "name": "шахтные воды", "matched_from": "запрос"},
    {"concept_id": "c_sulfate", "name": "сульфаты", "matched_from": "запрос"}
  ],

           "reason": "6 чанков прошли скор-гейт (≥2 сигналов или сильный семантический) ; сильное совпадение имени документа",
           "n_candidates": 41, "n_kept": 6, "namedoc": true},
  "docs_considered": 41
}
```
